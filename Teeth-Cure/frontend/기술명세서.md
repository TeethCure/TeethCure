## 도메인별 기능 명세서 (Functional Specification)

### 1. 🪥 양치 도메인 (Brushing Domain)

양치 가이드 제공, Vision AI를 통한 모션 인식, 실시간 상태 업데이트 및 최종 결과 처리를 담당하는 핵심 도메인입니다.

| **기능 ID** | **기능명** | **세부 설명** | **비고/정책** |
| --- | --- | --- | --- |
| **BR-01** | **양치 진입 가능 여부 확인** | 양치 모드 진입 전, 마지막 양치 시간으로부터 4시간이 경과했는지 확인합니다. | `User` 도메인의 쿨타임 데이터 연동 |
| **BR-02** | **가이드 제공** | 양치 시작 전 올바른 양치 가이드를 이미지로 넘길 수 있게 보여줍니다. | 다시는 보지 않기/닫기 기능 추가 |
| **BR-03** | **정밀 모션 및 위치 인식** | Vision AI를 통해 사용자의 치아 양옆 등 특정 위치를 잘 닦고 있는지 실시간으로 판별합니다. | 인식률에 따른 오차 보정 로직 필요 |
| **BR-04** | **캐릭터 정화 (진척도)** | 모션 인식이 성공적으로 이루어지는 동안, 화면 속 캐릭터를 덮는 거품이 증가합니다. | 닦는 부위에 따라 캐릭터에 거품이 추가되며 깨끗해지는 것을 시각화 |
| **BR-05** | **시간 게이지 증가** | 양치가 시작되면 제한 시간 게이지가 지속적으로 증가합니다. | 몰입도를 높이기 위해 초 단위 숫자보다 시각적 게이지로 표현 |
| **BR-06** | **양치 모습 사진 촬영** | 양치하는 모습을 캐릭터와 함께 담기도록 사진을 자동으로 찍어줍니다. | 양치 중 사진을 찍을 타이밍을 정하고, 얼굴 위주로 캐릭터 특징이 반영된 소품과 함께 사진 생성 |
| **BR-07** | **양치 결과 처리 (성공)** | 시간 게이지가 끝나고 캐릭터가 완전히 깨끗해지면 '구출 성공' 처리 및 다음 스토리를 해금합니다.  | 해당 캐릭터 획득 처리 및 `Character` 도메인 연동
사운드 제공 - 구출 성공 효과음  |
| **BR-08** | **양치 결과 처리 (실패)** | 캐릭터가 완전히 깨끗해지지 않은 상태에서 시간 게이지가 모두 채워지면 '구출 실패' 처리합니다. | 다음 스토리 확인 불가, 재도전 유도
캐릭터가 멀어지며 사라짐(발걸음 사운드)
사운드 제공 - 멀어지는 발걸음 효과음 |

### 기능 명세서: 1. 양치 세션 진입 (Start Brushing Session)

**대응 기능: BR-01(쿨타임 확인), BR-02(가이드 제공)**

**1. 개요 (Overview)**
사용자가 양치를 시작하기 전, 양치 가능 상태인지를 검증해서 진입하면, 양치 가이드 정보를 클라이언트에 반환하는 기능입니다.

**2. 입력 데이터 (Input Data)**
클라이언트로부터 별도의 Request Body는 받지 않습니다.

- **Authorization Header:** 로그인 시 발급받은 JWT 토큰 (여기서 `userId`를 추출)
- **Target Region ID (선택):** 사용자가 특정 구역을 선택해서 진입하는 경우 구역 ID.

**3. 상세 기능 요구사항 (Requirements)A. 유효성 검사 (Validation)**

- **인증 검증:** 유효한 JWT 토큰인지 확인 (만료 여부, 서명 검증).
- **쿨타임 검증 (Critical):** DB에서 해당 유저의 `last_brushing_at` 값을 조회하여, 현재 서버 시간 기준으로 4시간이 지났는지 엄격하게 검사.
    - 4시간이 지나지 않았다면 캐릭터나 스토리의 해금이 불가능하다는 경고 발생 및 남은 시간 반환.

**B. 비즈니스 로직 (Business Logic)**

- **타겟 캐릭터 할당:** 해당 구역에서 아직 구출하지 않은(도감에 없는) 캐릭터 중 하나를 랜덤 또는 순차적으로 선택하여 타겟으로 지정합니다.
- **초기 데이터 응답:** 선택된 캐릭터의 식별자, 이름, 그리고 클라이언트가 가이드를 띄우는 데 필요한 리소스 URL(가이드 영상 또는 이미지)을 반환합니다.

### 기능 명세서: 2. 양치 결과 제출 (Submit Brushing Result) ⭕

**대응 기능: BR-03, 04, 05(Vision AI 진행 결과), BR-06(성공 처리), BR-07(실패 처리)**

**1. 개요 (Overview)**
사용자가 양치를 마친 후(시간 게이지 또는 강제 종료), 클라이언트가 Vision AI를 통해 측정한 최종 정화도와 진행 시간을 서버로 전송하여 최종 성공/실패 여부를 확정하고 보상을 지급하는 기능입니다.

**2. 입력 데이터 (Input Data)**

- **durationSec (진행 시간):** 실제 양치를 진행한 시간(초).
- **purification (최종 정화도):** AI가 측정한 0, 1, 2, 3

**3. 상세 기능 요구사항 (Rrements)A. 유효성 검사 (Validation)**

- `purification`: 0 /1/2/3 정수여야 함
- 백으로는 성공 여부만 Boolean으로 보내기

**B. 비즈니스 로직 (Business Logic)**

- **성공/실패 판별 (Critical):**
    - `purification == 3`인 경우: **성공 (BR-06)** 처리. `UserCharacter` 테이블에 획득 로그 추가.
    - `purification < 3`인 경우: **실패 (BR-07)** 처리.
- **기록 저장:** 성공 여부와 관계없이 `BrushingLog` 테이블에 데이터를 Insert.
- **쿨타임 갱신:** 양치 결과와 무관하게(또는 정책에 따라 성공 시에만) `User` 테이블의 `last_brushing_at`을 현재 서버 시간으로 갱신.
- **트랜잭션 보장:** 로그 저장, 캐릭터 획득 처리, 유저 정보 업데이트가 하나의 `@Transactional`로 묶여야 합니다.부가 로직(알림 발송)

### 기능 명세서: 3. 캐릭터 및 도감 도메인 (Character & Collection Domain)

**1. 개요 (Overview)**
사용자가 성공적인 양치를 통해 구출한 캐릭터 목록을 확인하고, 해당 캐릭터에 얽힌 '정화 스토리'를 열람하는 기능입니다. 획득하지 못한 캐릭터는 실루엣(비활성화) 처리하여 수집 욕구를 자극합니다.

**2. 입력 데이터 (Input Data)**

- **Authorization Header:** 로그인 시 발급받은 JWT 토큰 (보안 컨텍스트에서 `userId` 추출)
- **Character ID (Path Variable):** 특정 캐릭터의 상세 스토리를 조회할 때 전달받는 식별자.

**3. 상세 기능 요구사항 (Requirements)**

**A. 유효성 검사 (Validation)**

- **조회 권한 검증 (Critical):** 캐릭터 상세 스토리를 요청할 때, 해당 `userId`가 `UserCharacter` 매핑 테이블에 해당 `characterId`를 실제로 보유하고 있는지 반드시 검증해야 합니다. 보유하지 않은 캐릭터의 스토리를 요청하면 `403 Forbidden` 예외를 발생시킵니다.
- **데이터 존재 여부:** 요청한 `characterId`가 `Character` 테이블에 존재하는지 확인 (`404 Not Found`).

**B. 비즈니스 로직 (Business Logic)**

- **도감 목록 조립:** * 전체 `Character` 목록을 조회합니다.
    - 현재 로그인한 사용자의 `UserCharacter` (획득 내역) 데이터를 교차 검증합니다.
    - 프론트엔드에서 그리기 쉽도록, 획득한 캐릭터는 `isUnlocked: true`와 함께 이미지 URL, 이름을 담아주고, 미획득 캐릭터는 `isUnlocked: false`로 처리하여 응답 데이터를 가공합니다.
- **스토리 해금 연출 데이터:** 캐릭터 상세 조회 시, 텍스트 형태의 스토리뿐만 아니라 프론트엔드에서 재생할 애니메이션/사운드 리소스 URL을 함께 반환합니다.
    - 데이터베이스에 저장되는 리소스 URL 형태
        
        데이터베이스에는 EC2 내부의 물리적 경로(예: `C:/` 또는 `/home/ubuntu/`)가 아니라, 프론트엔드가 외부에서 접근할 수 있는 **웹 URL** 형태의 문자열을 저장해야 합니다.
        
        - **저장 예시:** `http://{EC2_탄력적IP_또는_도메인}/media/characters/char_1_animation.mp4`
        - **사운드 예시:** `http://{EC2_탄력적IP_또는_도메인}/media/sounds/char_1_voice.mp3`
        
        클라이언트는 이 URL을 그대로 `<img>`, `<video>`, `<audio>` 태그의 `src` 속성에 넣어서 사용하게 됩니다.
        
    
    **💡 서연님을 위한 Deep Dive (기술적 고민)**
    
- **JPA N+1 문제:** 도감 목록을 불러올 때, 사용자가 보유한 `UserCharacter` 목록을 가져온 뒤 각 엔티티에 연결된 `Character` 정보를 지연 로딩(Lazy Loading)으로 가져오게 되면, 획득한 캐릭터 수만큼 추가 쿼리가 발생하는 전형적인 N+1 문제가 발생할 수 있습니다. 이를 방지하기 위해 Repository 계층에서 `JOIN FETCH`를 사용하거나 Entity Graph를 활용해 한 번의 쿼리로 연관 데이터를 가져오도록 최적화해야 합니다.

### 기능 명세서: 4. 월드맵 및 스토리 도메인 (World Map & Story Domain)

**1. 개요 (Overview)**
"세계를 정화하는 모험"이라는 핵심 세계관을 보여주는 지도 기능입니다. 각 구역(Region)의 정화 진행도를 확인하고, 구역 내 모든 캐릭터를 구출했을 때 활성화되는 구역별 메인 스토리를 제공합니다.

**2. 입력 데이터 (Input Data)**

- **Authorization Header:** 로그인 시 발급받은 JWT 토큰 (`userId` 추출)
- **Region ID (Path Variable):** 특정 구역의 상세 정보나 메인 스토리를 조회할 때 사용.

**3. 상세 기능 요구사항 (Requirements)A. 유효성 검사 (Validation)**

- **구역 스토리 열람 권한 검증 (Critical):** 특정 구역의 메인 스토리를 요청할 때, 사용자가 해당 구역에 속한 **모든 캐릭터를 구출했는지** 서버 측에서 다시 한번 깐깐하게 검증해야 합니다. 조건이 충족되지 않았다면 `403 Forbidden` 처리합니다.

**B. 비즈니스 로직 (Business Logic)**

- **진행도(Progress) 계산:** * 각 `Region`별로 할당된 총 캐릭터 수(`totalCount`)를 구합니다.
    - 사용자가 해당 `Region`에서 구출한 캐릭터 수(`unlockedCount`)를 카운트합니다.
    - `unlockedCount / totalCount` 비율을 프론트엔드에 전달하여 게이지바나 지도상의 시각적 효과를 제어할 수 있게 합니다.
- **구역 활성화 상태 판별:** `totalCount == unlockedCount`일 경우 해당 구역의 상태를 '정화 완료(활성화)'로 마킹하여 응답합니다.

**💡 서연님을 위한 Deep Dive (기술적 고민)**

- **관심사의 분리(AOP 적용 고민):** 사용자가 특정 스토리나 도감을 "처음으로 열람"했을 때 보상을 주거나 별도의 분석용 로그를 남겨야 하는 기획이 추가될 수 있습니다. 이때 핵심 비즈니스 로직에 로깅 코드가 섞이는 것을 막기 위해, Spring AOP를 활용하여 `@LogStoryAccess` 같은 커스텀 어노테이션을 만들고 프록시 객체를 통해 부가 기능을 분리해 내면 유지보수성이 훨씬 좋아집니다.
- **정적 데이터 캐싱:** `Region` 데이터나 `Character`의 기본 정보(이름, 소속 구역 등)는 자주 변경되지 않는 정적 성격의 데이터입니다. 사용자가 도감이나 지도를 열 때마다 DB를 조회하기보다, Redis나 Spring의 Local Cache(`@Cacheable`)를 적용해 두면 서버 응답 속도를 극적으로 끌어올릴 수 있습니다.

### 기능 명세서: 5. 내 정보 조회 (Get My Profile)

마이페이지 등에서 사용자(아이)의 서비스 이용 현황을 요약해서 보여주는 기능입니다. 아이가 얼마나 양치를 열심히 했는지 한눈에 파악하여 성취감을 느낄 수 있게 합니다.

**A. UI/UX 및 유효성 검사 (Validation)**

- 정보가 로딩되는 동안 사용자에게 로딩 스피너(또는 귀여운 대기 애니메이션)를 노출합니다.

**B. 핵심 서비스 로직 (Business Logic)**

- **기본 정보:** 사용자의 닉네임과 가입일을 표시합니다.
- **통계 정보:** '누적 양치 성공 횟수'와 '현재까지 구출한 총 캐릭터 수(도감 수집률)'를 계산하여 화면에 표시합니다.

**💡 파트별 협업 포인트 (Collaboration Points)**

- **기획/디자인:** 수집률이나 양치 횟수에 따라 칭호(예: "양치 초보", "정화의 용사")를 부여하는 UI를 추가할지 고려해 보세요.
- **프론트엔드:** 통계 숫자가 올라가는 카운팅 애니메이션을 적용하면 아이들의 흥미를 더 끌 수 있습니다.
- **백엔드:** 누적 데이터를 매번 전체 기록에서 세는 대신, 정보가 갱신될 때마다 요약 테이블에 업데이트해 두면 조회 속도를 높일 수 있습니다.

### 기능 명세서: 6. 양치 쿨타임 조회 (Check Cooldown)

**1. 개요 (Overview)**

아이들이 양치를 연속으로 하거나 장난치는 것을 방지하고, 올바른 양치 주기를 학습하도록 돕기 위해 '마지막 양치 후 4시간'이 지났는지 확인하는 제어 기능입니다.

**2. 필요 데이터 및 발생 조건 (Input / Conditions)**

- **발생 조건:** 메인 화면 진입 시 또는 '양치 시작' 버튼 노출 시점
- **필요 데이터:** 해당 사용자의 마지막 양치 완료 시간

**3. 상세 기능 요구사항 (Requirements)**

**A. UI/UX 및 유효성 검사 (Validation)**

- 4시간이 지나지 않았다면 캐릭터나 스토리를 해금할 수 없습니다.
- 남은 시간을 직관적으로(예: "02:15:30 후 해금 가능" 또는 시각적인 타이머) 표시합니다.

**B. 핵심 서비스 로직 (Business Logic)**

- 현재 시간과 사용자의 마지막 양치 완료 시간을 비교합니다.
- **4시간 경과 시:** 양치 모드 진입을 허용합니다.
- **4시간 미경과 시:** 캐릭터나 스토리가 해금이 불가능하다는 경고 문구를 표시하고, 쿨타임이 해제되기까지 남은 시간을 계산하여 화면에 실시간으로 줄어들게 표시합니다.

**💡 파트별 협업 포인트**

- **기획/디자인:** 쿨타임 중에 아이들이 앱에 들어왔을 때 실망하지 않도록, 도감이나 지도를 구경하도록 유도하는 동선 설계가 필요합니다.
- **프론트/백엔드:** 기기 시간을 임의로 조작(예: 스마트폰 시간 설정 변경)하는 어뷰징을 막기 위해, 시간 계산의 기준은 반드시 '서버 시간'이어야 합니다.

### 기능 명세서: 7. 양치 세션 초기화 (Initialize Session)

**1. 개요 (Overview)**

쿨타임이 지나 양치를 시작하려는 아이에게, 이번 양치에서 구출할 '목표 캐릭터'를 배정하고 화면에 띄워줄 가이드 영상을 준비하는 단계입니다.

**2. 필요 데이터 및 발생 조건 (Input / Conditions)**

- **발생 조건:** '양치 시작' 버튼 클릭 시
- **필요 데이터:** 사용자의 이전 캐릭터 획득 내역 (중복 구출 방지용)

**3. 상세 기능 요구사항 (Requirements)**

**A. UI/UX 및 유효성 검사 (Validation)**

- 카메라 권한이 허용되어 있는지 검사하고, 미허용 시 권한 요청 팝업을 띄웁니다.
- 진입 직전에 쿨타임(4시간)을 한 번 더 최종 확인하여 오류를 방지합니다.

**B. 핵심 서비스 로직 (Business Logic)**

- 사용자가 아직 구출하지 못한(도감에 없는) 캐릭터 중 하나를 이번 양치의 타겟으로 선정합니다.
- 선정된 타겟 캐릭터의 기본 정보(이름)와 양치 가이드를 시연할 미디어(영상 또는 애니메이션) 데이터를 로드합니다.
- 비정상적인 결과 제출을 막기 위해 이번 양치 활동에 대한 고유한 세션(티켓)을 발급합니다.

**💡 파트별 협업 포인트**

- **기획:** 만약 사용자가 모든 캐릭터를 다 구출했다면 어떤 캐릭터를 타겟으로 줄지(랜덤 등장, 경험치만 제공 등) 예외 정책을 정해야 합니다.
- **프론트엔드:** 카메라 화면과 가이드 영상이 겹쳐 보일 때 이질감이 없도록 UI 레이아웃 배치가 중요합니다.

### 기능 명세서: 8. 실시간 양치 모션 인식 (Vision AI)

**1. 개요 (Overview)**

스마트폰 전면 카메라를 이용해 아이가 양치하는 모습을 실시간으로 인식하고, 양치를 잘할수록 화면 속 캐릭터의 오염도가 씻겨 나가는 핵심 플레이 기능입니다.

**2. 필요 데이터 및 발생 조건 (Input / Conditions)**

- **발생 조건:** 양치 세션 진입 직후
- **필요 데이터:** 카메라 실시간 영상 프레임

**3. 상세 기능 요구사항 (Requirements)**

**A. UI/UX 및 유효성 검사 (Validation)**

- 얼굴이 화면 밖으로 나가거나 칫솔이 안 보일 경우, "얼굴을 보여주세요!" 같은 시각적/~~청각적~~ 피드백을 주어 자세를 교정하도록 유도합니다.

**B. 핵심 서비스 로직 (Business Logic)**

- 화면 한 쪽에 캐릭터 가이드가 재생되며 올바른 양치 동작을 시연합니다.
- **AI 인식:** 카메라 영상을 분석하여 칫솔의 위치(좌, 우, 앞)와 움직임을 판별합니다.
- **오염 정화 및 타이머:** 인식률에 따라 화면 속 타겟 캐릭터를 덮고 있는 오염 효과(시각 효과)를 점진적으로 제거하고, 정화 게이지를 채웁니다. 
동시에 남은 시간(타이머)은 줄어듭니다.
- 타이머가 0이 되거나 정화 게이지가 100%가 되면 즉시 인식을 멈추고 결과를 산출합니다.

**💡 파트별 협업 포인트**

- **AI/프론트엔드:** 스마트폰에서 실시간으로 AI 모델이 돌아가야 하므로, 기기 발열이나 버벅임(프레임 드랍)을 최소화하는 모델 경량화가 매우 중요합니다.
- **디자인:** 아이들이 게이지가 차오르는 것을 직관적으로 느낄 수 있도록 화려한 파티클 효과나 사운드를 적극 활용해야 합니다.

### 기능 명세서: 9. 양치 결과 기록 및 제출 (Submit Result)

**1. 개요 (Overview)**

양치 제한 시간이 종료된 후, 아이가 얼마나 양치를 잘했는지(정화 게이지)를 판별하여 최종적으로 캐릭터 구출 성공/실패 여부를 확정 짓는 기능입니다.

**2. 필요 데이터 및 발생 조건 (Input / Conditions)**

- **발생 조건:** 양치 제한 시간이 0초가 되거나 정화 게이지가 100% 달성 시
- **필요 데이터:** 최종 정화 게이지 수치(0~100%), 실제 진행 시간

**3. 상세 기능 요구사항 (Requirements)**

**A. UI/UX 및 유효성 검사 (Validation)**

- 통신이 진행되는 동안 화면이 멈춘 것처럼 보이지 않도록 결과 산출 중임을 알리는 연출을 제공합니다.
- 비정상적인 수치(예: 3초 만에 완료)는 무효 처리합니다.

**B. 핵심 서비스 로직 (Business Logic)**

- **공통:** 양치 시간, 달성 게이지 등을 서비스에 기록하고 마지막 양치 시간을 지금으로 갱신합니다.
- **성공 처리 (정화 완료 시):**
    - 타겟 캐릭터 구출 성공으로 판정하고, 사용자의 도감에 해당 캐릭터를 추가합니다.
    - 캐릭터가 맑아지는 연출과 함께 해금된 스토리(축하 메시지)를 화면에 표시합니다.
- **실패 처리 (시간 초과 시 정화 미달성):**
    - 캐릭터 구출에 실패하며 도감에 추가되지 않습니다.
    - "조금만 더 닦으면 구할 수 있었어! 다음엔 더 잘해볼까?" 등의 격려 메시지를 노출합니다.

**💡 파트별 협업 포인트**

- **프론트/백엔드:** 화장실에서 양치를 끝냈을 때 와이파이가 끊기는 경우가 잦습니다. 실패 시 기기 내부에 잠시 저장했다가 네트워크가 복구되면 결과를 전송하는 로직이 있으면 좋습니다.
- **기획/디자인:** 실패했을 때 아이가 좌절감을 느끼지 않게끔 긍정적인 UX 라이팅(UX Writing)과 캐릭터 표정을 준비해야 합니다.

### 기능 명세서: 10. 월드맵 전체 상태 조회 (Get World Map Status)

**1. 개요 (Overview)**

'세계를 정화하는 모험'이라는 컨셉을 시각적으로 보여주기 위해, 전체 지도를 띄우고 각 구역(숲, 바다 등)이 얼마나 정화되었는지 상태를 보여주는 기능입니다.

**2. 필요 데이터 및 발생 조건 (Input / Conditions)**

- **발생 조건:** 하단 탭 메뉴에서 '월드맵/지도' 메뉴 클릭 시
- **필요 데이터:** 전체 구역 목록과 구역별 요구 캐릭터 수, 사용자의 캐릭터 보유 현황

**3. 상세 기능 요구사항 (Requirements)**

**A. UI/UX 및 유효성 검사 (Validation)**

- 아직 정화되지 않은 구역은 어둡거나 오염된 이미지로, 정화가 완료된 구역은 밝고 깨끗한 이미지로 구분해서 보여줍니다.

**B. 핵심 서비스 로직 (Business Logic)**

- 지도에 존재하는 전체 구역 리스트를 불러옵니다.
- 각 구역에 속한 전체 캐릭터 수와, 아이가 해당 구역에서 구출한 캐릭터 수를 비교합니다.
- 해당 구역의 캐릭터를 모두 구출했다면 해당 구역을 '정화 완료(활성화)' 상태로 처리합니다.

**💡 파트별 협업 포인트**

- **디자인:** 지도가 단순히 버튼의 나열이 되지 않도록, 실제 모험을 하는 듯한 유려한 일러스트와 구역 간 이어지는 길(Path) 디자인이 필요합니다.
- **프론트엔드:** 지도를 스와이프하거나 확대/축소할 수 있는 뷰어(Viewer) 기능 구현 여부를 기획과 논의해야 합니다.

### 기능 명세서: 11. 특정 구역 상세 및 메인 스토리 조회 (Get Region Detail & Story)

**1. 개요 (Overview)**

지도에서 하나의 구역을 선택했을 때, 해당 구역을 완전히 정화하기까지 얼마나 남았는지 진행도를 보여주고, 100% 정화 시 숨겨진 큰 이야기를 열람하는 기능입니다.

**2. 필요 데이터 및 발생 조건 (Input / Conditions)**

- **발생 조건:** 월드맵에서 특정 구역(핀 또는 일러스트)을 터치할 때
- **필요 데이터:** 선택한 구역의 식별 정보

**3. 상세 기능 요구사항 (Requirements)**

**A. UI/UX 및 유효성 검사 (Validation)**

- 정화가 100% 완료되지 않은 상태에서 스토리를 열람하려고 하면 자물쇠 아이콘 등을 띄워 접근을 차단합니다.

**B. 핵심 서비스 로직 (Business Logic)**

- **진행도 표시:** "현재 구출한 캐릭터 수 / 구역 내 전체 캐릭터 수"를 계산해 게이지 바(%) 형태로 표시합니다.
- **스토리 해금:** 진행도가 100%일 경우, 해당 구역이 깨끗해지면서 벌어지는 메인 스토리(일러스트 기반 컷신 또는 텍스트)를 노출합니다.

**💡 파트별 협업 포인트**

- **기획:** 구역 스토리를 한 번 보고 끝낼지, 언제든 다시 볼 수 있도록 할지 리플레이 정책을 정해야 합니다.
- **디자인:** 구역별 스토리에 몰입할 수 있도록 각 구역의 테마에 맞는 배경음악(BGM)이 깔리면 좋습니다.

### 기능 명세서: 12. 전체 캐릭터 도감 조회 (Get Character Collection)

**1. 개요 (Overview)**

아이들이 양치 보상으로 수집한 캐릭터 전체 리스트를 한곳에 모아 보여주어, 수집 욕구와 성취감을 자극하는 도감 메인 화면입니다.

**2. 필요 데이터 및 발생 조건 (Input / Conditions)**

- **발생 조건:** 메인 메뉴에서 '도감/캐릭터' 메뉴 클릭 시
- **필요 데이터:** 사용자의 캐릭터 구출(획득) 정보

**3. 상세 기능 요구사항 (Requirements)**

**A. UI/UX 및 유효성 검사 (Validation)**

- 도감은 리스트 또는 그리드(격자) 형태로 제공됩니다.
- 구출하지 못한 캐릭터는 검은 실루엣이나 물음표(?) 등으로 렌더링하여 호기심을 유발합니다.

**B. 핵심 서비스 로직 (Business Logic)**

- 서비스 내 존재하는 모든 캐릭터 목록을 불러와 사용자의 보유 기록과 대조합니다.
- **획득 캐릭터:** 원래의 예쁜 이미지와 이름을 노출합니다.
- **미획득 캐릭터:** 상태를 비활성화(Locked)로 처리하고 상세 정보 접근을 막습니다.

**💡 파트별 협업 포인트**

- **프론트엔드:** 캐릭터 일러스트 이미지가 많아지면 로딩이 느려질 수 있습니다. 스크롤을 내릴 때 이미지를 불러오는 지연 로딩(Lazy Loading) 처리가 필요합니다.

### 기능 명세서: 13. 특정 캐릭터 상세 및 숨겨진 스토리 조회 (Get Character Detail)

**1. 개요 (Overview)**

도감에서 내가 구출한 특정 캐릭터를 눌렀을 때, 해당 캐릭터의 크고 선명한 일러스트와 고유한 이야기(설정)를 보여주는 기능입니다.

**2. 필요 데이터 및 발생 조건 (Input / Conditions)**

- **발생 조건:** 도감 화면에서 활성화된 특정 캐릭터 터치 시
- **필요 데이터:** 선택한 캐릭터 식별 정보

**3. 상세 기능 요구사항 (Requirements)**

**A. UI/UX 및 유효성 검사 (Validation)**

- 미획득 캐릭터를 비정상적인 방법으로 터치하려 할 경우 화면 진입을 막습니다.
- 캐릭터의 애니메이션(있는 경우)을 반복 재생하여 생동감을 줍니다.

- 캐릭터의 기본 정보(이름, 소속 구역 특징 등)를 제공합니다.

**💡 파트별 협업 포인트**

- **기획/디자인:** 포켓몬스터 도감처럼 캐릭터별 키/몸무게/좋아하는 치약 맛 같은 아기자기한 설정을 텍스트로 넣어두면 아이들이 훨씬 더 좋아합니다.
- **프론트엔드:** 페이지 전환 시 캐릭터 이미지가 자연스럽게 커지는 트랜지션(Hero Animation)을 넣으면 사용성이 아주 좋아 보입니다.

### 기능 명세서: 14. 회원가입 (Sign Up)

**1. 개요 (Overview)**

서비스 이용을 위해 새로운 계정을 생성하는 기능입니다. 부모님(또는 보호자)이 아이의 계정을 만들어주는 흐름을 가정하며, 민감한 개인정보인 비밀번호는 안전하게 암호화되어 데이터베이스에 저장됩니다.

**2. 필요 데이터 및 발생 조건 (Input / Conditions)**

- **발생 조건:** 앱 실행 후 최초 진입 시 또는 '회원가입' 버튼 클릭 시
- **필요 데이터:** 사용자가 입력한 이메일, 비밀번호, 아이 닉네임

**3. 상세 기능 요구사항 (Requirements)**

**A. UI/UX 및 유효성 검사 (Validation)**

- **아이디:** 중복 가입이 안 되게 검사합니다.
- **비밀번호:** 최소 8자 이상, 영문/숫자/특수문자 포함 등 보안 정책을 적용하여 입력창 하단에 충족 여부를 실시간으로 표시합니다.
- 이름: 부모의 실명을 입력합니다.
- 생년월일: 자녀의 생년월일을 입력합니다.

**B. 핵심 서비스 로직 (Business Logic)**

- **이메일 중복 검사:** 이미 가입된 이메일인지 데이터베이스를 조회하여, 중복 시 가입을 차단하고 안내 메시지를 띄웁니다.
- **비밀번호 단방향 암호화:** 평문(Plain Text) 비밀번호는 절대 그대로 저장하지 않습니다. 별도의 암호화 라이브러리(예: jBCrypt)를 사용하여 단방향 해싱 처리 후 데이터베이스에 저장합니다.
- **초기 상태 세팅:** 가입 시 양치 쿨타임 계산을 위한 마지막 양치 시간은 초기값(Null 등)으로 설정합니다.

**💡 파트별 협업 포인트**

- **프론트엔드/기획:** 이메일 중복 확인을 '가입 완료' 버튼을 누를 때 할지, 이메일 입력 칸 옆에 '중복 확인' 버튼을 따로 둘지 UX 정책을 결정해야 합니다.
- **백엔드:** 찰나의 순간에 동일한 이메일로 두 번의 요청이 들어오는 동시성 문제를 막기 위해, 데이터베이스의 이메일 컬럼에 반드시 Unique Index(고유 제약조건)를 설정해야 합니다.

### **기능 명세서: 15. 로그인 및 자동 로그인 (Login & Auto-Login)**

**1. 개요 (Overview)**
사용자가 이메일과 비밀번호로 최초 인증을 거치면 유효기간이 긴 '단일 출입증(Access Token)'을 발급받아 기기에 저장하고, 이후 앱을 켤 때마다 이 출입증을 사용해 별도의 로그인 과정 없이 서비스를 바로 이용하게 해주는 기능입니다.

**2. 필요 데이터 및 발생 조건 (Input / Conditions)**

- **발생 조건 1 (최초 로그인):** 로그인 화면에서 이메일과 비밀번호 입력 후 '로그인' 버튼 클릭 시
- **발생 조건 2 (자동 로그인):** 앱 아이콘을 눌러 앱을 실행할 때 (스플래시 화면 대기 중)
- **필요 데이터:**
    - 최초 로그인: 사용자가 입력한 이메일 주소, 평문 비밀번호
    - 자동 로그인: 기기 내부(보안 저장소)에 보관된 유효한 Access Token

**3. 상세 기능 요구사항 (Requirements)A. UI/UX 및 유효성 검사 (Validation)**

- **최초 로그인 실패 시:** "아이디 또는 비밀번호가 일치하지 않습니다"라는 통합 안내 메시지를 노출합니다.
- **자동 로그인 처리:** 앱 실행 시 스플래시(로고) 화면을 띄워두고 백그라운드에서 토큰 유효성을 검사합니다.
    - 검증 성공(토큰 유효): 로그인 화면을 건너뛰고 곧바로 메인 화면(월드맵 등)으로 진입합니다.
    - 검증 실패(토큰 없음 또는 만료): 사용자에게 로그인 화면을 노출하여 재로그인을 유도합니다.

**B. 핵심 서비스 로직 (Business Logic)**

- **계정 및 비밀번호 검증:** 데이터베이스에서 이메일로 사용자를 조회한 뒤, 입력받은 비밀번호를 암호화하여 저장된 암호문과 비교합니다.
- **장기 JWT 토큰 발급:** 인증 성공 시, 자동 로그인 유지를 위해 유효기간이 매우 긴(예: 1년) Access Token 1개만 단독으로 발급하여 클라이언트에 전달합니다.
- **자동 로그인 통신:** 앱 실행 시 클라이언트가 보관 중인 Access Token을 담아 내 정보 조회 등의 기본 API를 찔러보고, 정상 응답이 오면 자동 로그인을 승인합니다.

**💡 파트별 협업 포인트**

- **프론트엔드 (앱 개발):** 토큰의 수명이 매우 길기 때문에 보안이 중요합니다. 모바일 OS가 제공하는 안전한 보안 저장소(iOS: Keychain, Android: Keystore)에 토큰을 보관해야 합니다.
- **백엔드:** 토큰 재발급(Reissue) 로직이나 Redis를 활용한 복잡한 상태 관리가 필요 없어 서버 구현이 매우 가벼워집니다. 모든 API 앞단에서 해당 JWT의 서명과 만료 여부만 검사하면 됩니다.
